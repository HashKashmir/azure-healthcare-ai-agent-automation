# Mock APIs & Synthetic Data

This document explains the two mock APIs (PrimeHR and ClaimBridge), the data generation system, and all the synthetic datasets used in this project. Every piece of data used here is fake — generated by a Python library called Faker combined with hand-crafted logic to make it realistic.

---

## Why Mock APIs?

In a real healthcare organization, this project would connect to systems like:
- **Workday or Oracle HCM** — for HR employee data (Automation 1)
- **Epic, Athenahealth, or eClinicalWorks** — for patient intake (Automation 2)
- **Athenahealth Billing, Change Healthcare, or Waystar** — for billing claims (Automations 2/3)

These are enterprise systems with access control, licensing costs, compliance requirements, and complex APIs. For a portfolio project, we simulate their core behaviors with two FastAPI-based mock servers that expose realistic data through REST APIs.

The mock APIs are not shortcuts or simplifications of the architecture — they're a deliberate design that makes the project more realistic (the tools still make real HTTP requests and parse real JSON responses) while remaining self-contained.

## PrimeHR Mock API — `mock_apis/primehr_api.py`

### What It Simulates

PrimeHR is the fictitious HR Information System (HRIS) for this healthcare organization. In the real world, this would be Workday, ADP Workforce Now, or Oracle HCM Cloud.

### How It Runs

```bash
uvicorn mock_apis.primehr_api:app --port 8001
```

Or automatically when you start the dashboard (dashboard.py launches it as a subprocess).

Interactive API docs: `http://localhost:8001/docs`

### Data Loading

On startup, the API loads `mock_apis/data/employees.json` into memory:

```python
DATA_FILE = os.path.join(os.path.dirname(__file__), "data", "employees.json")

def _load_employees() -> list[dict]:
    with open(DATA_FILE, encoding="utf-8") as f:
        return json.load(f)

_employees: list[dict] = _load_employees()
_by_id: dict[str, dict] = {e["employee_id"]: e for e in _employees}
```

All 75 employees are loaded into memory once at server start. All requests are served from the in-memory dictionary — no database needed. Restarting the server reloads fresh data from the JSON file.

### Endpoints

**`GET /`** — API info
```json
{
  "api": "PrimeHR HR API (Mock)",
  "version": "1.0.0",
  "description": "Serves synthetic employee records for Healthcare AI agent integration."
}
```

**`GET /health`** — health check (used by the dashboard to show the green status dot)
```json
{"status": "healthy", "timestamp": "2025-05-25T14:32:00Z", "employee_count": 75}
```

**`GET /employees`** — list all employees with optional filters:
- `?department=Cardiology` — filter by department
- `?status=Active` — filter by active/inactive status
- `?onboarding_status=In Progress` — filter by onboarding completion

**`GET /employees/{employee_id}`** — fetch one employee's full record. Used by `get_employee_details` and `fill_onboarding_form`. Returns the complete employee dictionary from `employees.json`.

**`GET /employees/{employee_id}/onboarding`** — returns the onboarding checklist status. This endpoint dynamically computes the completed vs. pending documents from the employee's `documents_pending` field:

```python
all_docs = [
    "W-4 Federal Withholding", "AZ State Tax Form", "I-9 Employment Eligibility",
    "Direct Deposit Authorization", "Benefits Enrollment Form",
    "HIPAA Confidentiality Agreement", "Employee Handbook Acknowledgment",
    "Background Check Consent", "Drug Screen Authorization",
    "Emergency Contact Form", "IT Access Request",
]
completed = [d for d in all_docs if d not in emp["documents_pending"]]
pending   = emp["documents_pending"]

return {
    "employee_id":       employee_id,
    "onboarding_status": emp["onboarding_status"],
    "checklist": {
        "completed_documents": completed,
        "pending_documents":   pending,
    }
}
```

**`POST /employees/{employee_id}/onboarding/complete`** — marks a specific document step as completed. Accepts `{document_name, completed_by, notes}`. Removes the document from `documents_pending` in memory. This mutation is lost on server restart.

## ClaimBridge Mock API — `mock_apis/claimbridge_api.py`

### What It Simulates

ClaimBridge is the fictitious billing claims management system. In the real world, this would be Athenahealth's billing module, Change Healthcare, Waystar (formerly Navicure), or Epic's revenue cycle module.

### How It Runs

```bash
uvicorn mock_apis.claimbridge_api:app --port 8002
```

Interactive API docs: `http://localhost:8002/docs`

### Data Loading

```python
DATA_FILE = os.path.join(os.path.dirname(__file__), "data", "claims.csv")

def _load_claims() -> list[dict]:
    with open(DATA_FILE, newline="", encoding="utf-8") as f:
        reader = csv.DictReader(f)
        rows = []
        for row in reader:
            row["billed_amount"]  = float(row["billed_amount"])
            row["allowed_amount"] = float(row["allowed_amount"])
            row["days_in_queue"]  = int(row["days_in_queue"])
            rows.append(row)
        return rows

_claims: list[dict] = _load_claims()
_by_id: dict[str, dict] = {c["claim_id"]: c for c in _claims}
```

Note the type casting: CSV files store everything as strings. The API casts numeric fields back to `float` and `int` on load.

### Endpoints

**`GET /claims/exceptions`** — the most important endpoint. Returns only `exception` and `denied` status claims, sorted by priority (critical first), limited to the requested count:

```python
priority_order = {"critical": 0, "high": 1, "medium": 2, "low": 3, "": 4}
exceptions = [c for c in _claims if c["claim_status"] in ("exception", "denied")]
exceptions.sort(key=lambda c: priority_order.get(c.get("priority", ""), 4))
return {"exceptions": exceptions[:limit], "queue_depth": len(exceptions)}
```

**`GET /claims/{claim_id}`** — full claim record. Used by `get_claim_details`. Returns the complete claim dictionary.

**`POST /claims/{claim_id}/resubmit`** — simulates resubmission. Mutates the in-memory claim:

```python
claim["claim_status"]     = "resubmitted"
claim["resolution_notes"] = body.resolution_notes or "Auto-resolved."
if body.corrected_icd10_code:
    claim["icd10_code"] = body.corrected_icd10_code
if body.prior_auth_number:
    claim["prior_auth_number"] = body.prior_auth_number
claim["resubmitted_at"] = datetime.utcnow().isoformat() + "Z"
```

**`POST /claims/{claim_id}/route`** — simulates routing to staff:

```python
claim["claim_status"] = "routed"
claim["assigned_to"]  = body.assigned_to
claim["priority"]     = body.priority
claim["routing_reason"] = body.routing_reason
claim["routed_at"]    = datetime.utcnow().isoformat() + "Z"
```

**`PATCH /claims/{claim_id}/status`** — general status update.

**`GET /claims/stats`** — aggregate statistics across all claims (total counts by status, average billed/allowed amounts, etc.). Used for dashboard status displays.

## Data Generation — `mock_apis/generate_data.py`

This script generates all three synthetic datasets. Run it once before starting the project:

```bash
python mock_apis/generate_data.py
```

It produces:
- `mock_apis/data/employees.json` — 75 employee records
- `mock_apis/data/claims.csv` — 300 billing claim records
- `data/weekly_metrics.csv` — 120 rows of operational metrics

### The Faker Library

```python
from faker import Faker
fake = Faker()
random.seed(42)
Faker.seed(42)
```

Faker generates realistic-looking fake data: names, addresses, phone numbers, emails. The `random.seed(42)` and `Faker.seed(42)` make the output deterministic — running the script twice produces identical data. This is important for reproducibility: your test data is the same every time you regenerate it.

## Employee Data — `generate_employees(n=75)`

### Structure

75 employees across 10 departments:
- Cardiology, Orthopedics, Pediatrics, Neurology, Oncology
- Emergency Medicine, Radiology, Physical Therapy, Mental Health, Primary Care

### Two-Pass Generation

The generator uses two passes to ensure every department has a manager before regular employees are assigned to managers:

**Pass 1 — Create 10 managers (one per department):**
```python
for i, dept in enumerate(DEPARTMENTS):
    mgr_first = fake.first_name()
    mgr_last  = fake.last_name()
    mgr_id    = f"EMP-{(i + 1):04d}"  # EMP-0001 through EMP-0010
    managers.append({
        "employee_id": mgr_id,
        "department":  dept,
        ...
    })
```

**Pass 2 — Create 65 regular employees:**
```python
for i in range(65):
    dept   = random.choice(DEPARTMENTS)
    pos    = random.choice(POSITIONS_BY_DEPT[dept])
    mgr    = next(m for m in managers if m["department"] == dept)

    emp_id = f"EMP-{(len(managers) + i + 1):04d}"  # EMP-0011 through EMP-0075
    ...
    employee["manager_id"]    = mgr["employee_id"]
    employee["manager_name"]  = f"{mgr['first_name']} {mgr['last_name']}"
    employee["manager_email"] = mgr["email"]
```

This ensures every employee has a valid manager with a real email address, and every department has exactly one manager.

### Position-Based Pay Ranges

PAY_RANGES = {

This makes the data realistic — a cardiologist earns $280k–$420k, a medical assistant earns $35k–$50k. These ranges are based on national healthcare salary benchmarks.

## Billing Claims Data — `generate_claims(n=300)`

### Structure

300 claims across all 10 departments with a status distribution weighted to create a realistic mix:

```python
STATUSES = ["approved", "approved", "approved", "pending", "pending",
            "exception", "exception", "denied", "resubmitted", "closed"]
```

This weight gives approximately:
- 30% approved
- 20% pending
- 20% exception/denied (the queue the billing agent works from)
- 10% resubmitted
- 10% closed

Of the 300 claims, about 60–100 end up in `exception` or `denied` status — the billing exception queue.

## Weekly Metrics Data — `generate_weekly_metrics(weeks=12)`

### Structure

120 rows (12 weeks × 10 departments) with 16 columns covering financial, clinical, and billing metrics.

### Realistic Variance

```python
def generate_weekly_metrics(weeks: int = 12) -> list[dict]:
    for w in range(weeks):
        week_start = (base_date + timedelta(weeks=w)).strftime("%Y-%m-%d")
        for dept in DEPARTMENTS:
            trend = 1 + (w * 0.005)         # slight upward trend (+0.5% per week)
            noise = random.uniform(0.88, 1.12)  # ±12% random noise

            visits       = int(random.randint(80, 220) * trend * noise)
            no_shows     = int(visits * random.uniform(0.07, 0.22))  # 7–22% no-show rate
            ...
            total_revenue = round(visits * billed_per_visit * trend * noise, 2)
            collections   = round(total_revenue * collection_rate, 2)
```

The `trend × noise` multiplication gives each week a slight overall improvement while maintaining realistic week-to-week variability. The no-show rate ranges from 7–22% across departments and weeks — some departments have chronic no-show problems (above the 10% benchmark), others don't.

### Why 10 Departments × 12 Weeks?

The reporting pipeline filters to the 4 most recent weeks. Having 12 weeks of history means:
- The trend analysis (4-week trend: improving/stable/worsening) has 4 genuine data points
- A new run of generate_data doesn't immediately show the same 4 weeks — the "recent" 4 weeks shift each time you regenerate based on today's date
- Charts can show all 4 weeks of history, not just the current vs. prior week

## Data Templates — `data/bulk_onboarding_template.xlsx` and `data/bulk_intake_template.xlsx`

### Bulk Onboarding Template

An Excel file with columns: `employee_id, first_name, last_name, department`

Pre-populated with 5 employee IDs from `employees.json` (EMP-0011 through EMP-0015). The dashboard's bulk upload endpoint reads the `employee_id` column and ignores the rest (the rest is for human readability — the agent fetches all employee data from the PrimeHR API anyway).

### Bulk Intake Template

An Excel file with columns: `first_name, last_name, dob, gender, address, phone, email, ssn_last4, insurance_id, insurance_name, group_number, provider, referring, complaint, department, visit_type, ec_name, ec_phone, ec_relationship`

Pre-populated with 5 patients, intentionally including one with `insurance_id=INS-005` (the terminated Humana HMO plan). This demonstrates the ineligibility handling workflow in bulk mode — you see 4 green "Eligible" cards and 1 red "Ineligible" card when you process the template.

The dashboard's bulk intake upload endpoint reads every column and passes all fields to the bulk intake runner as patient dictionaries, which then uses the data to generate per-patient PDFs and JSON files.

## File Exclusions from Git

Several data files are excluded from git via `.gitignore`:

```
data/weekly_metrics.csv          # regenerated each time (date-sensitive)
data/onboarding_docs/            # generated PDFs from agent runs
data/reports/                    # generated HTML/PDF reports
data/report_charts/              # generated chart PNGs
data/intake_patient_data.json    # generated during intake runs
data/intake_patient_*.json       # per-patient data files (bulk intake)
data/intake_bulk_*.pdf           # per-patient PDFs (bulk intake)
data/bulk_intake_patients.json   # written during bulk upload
audit/audit_log.jsonl            # run history with entity IDs
```

The committed files are only the static templates and the pre-generated sample:
- `data/sample_intake.pdf` — static, pre-generated
- `data/bulk_onboarding_template.xlsx` — static template
- `data/bulk_intake_template.xlsx` — static template
- `mock_apis/data/employees.json` — static synthetic data (deterministic)
- `mock_apis/data/claims.csv` — static synthetic data (deterministic)

Because `generate_data.py` uses fixed random seeds (`random.seed(42)`, `Faker.seed(42)`), the employees and claims data is identical every time you regenerate — so it's safe to commit the generated files.

`weekly_metrics.csv` is excluded because it uses dates relative to `datetime.now()`. The "most recent 4 weeks" would drift over time if the same CSV was committed, making the report show stale weeks. Regenerating it fresh ensures the data always covers the actual recent weeks.
